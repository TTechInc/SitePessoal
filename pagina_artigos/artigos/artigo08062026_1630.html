Geração dos Programadores Superficiais

Pedro Oliveira · 08/06/2026

Nos últimos anos, a informática sofreu uma transformação curiosa. Cada vez mais pessoas entram na área motivadas pela programação, mas acabam por reduzir toda a informática à programação. O resultado é uma geração de profissionais que conhece várias linguagens, frameworks e bibliotecas, mas possui um conhecimento limitado dos fundamentos que sustentam os sistemas computacionais. Hoje é comum encontrar quem saiba desenvolver aplicações web complexas, utilizar dezenas de bibliotecas e trabalhar com tecnologias modernas, mas tenha dificuldade em explicar conceitos básicos sobre memória, sistemas operativos, arquiteturas de processador, redes ou compiladores. Este fenómeno não é necessariamente culpa dos estudantes. Em grande parte, resulta da forma como a informática é ensinada, promovida e vendida ao público.

A obsessão pelas linguagens

Um dos sintomas mais visíveis desta tendência é a obsessão pelas linguagens de programação. Muitos iniciantes acreditam que aprender informática significa aprender linguagens. Começam por Python, depois JavaScript, seguem para Java, C#, Go, Rust ou qualquer outra tecnologia popular do momento. A pergunta recorrente é sempre:Que linguagem devo aprender a seguir? Raramente a pergunta é: Como funciona a memória virtual? Como é implementado um sistema operativo? O que acontece quando um programa é compilado? Como é executada uma instrução dentro de um processador? A consequência é a criação da ilusão de que o conhecimento informático é proporcional ao número de linguagens conhecidas. Na realidade, as linguagens são apenas ferramentas para expressar algoritmos e interagir com sistemas. Os conceitos fundamentais permanecem os mesmos.

A ilusão de que Assembly é apenas mais uma linguagem

Esta mentalidade torna-se particularmente evidente quando alguém tenta aprender Assembly. Muitos estudantes olham para Assembly da mesma forma que olham para Python ou Java: Aprender sintaxe, fazer exercícios, decorar instruções, escrever programas. Mas Assembly não é apenas uma linguagem. Assembly é uma representação quase direta da arquitetura do processador. Quando alguém tenta aprender Assembly sem compreender: sistemas digitais, álgebra booleana, organização de computadores, registos, memória, ciclo de execução de instruções, arquitetura do processador, o estudo torna-se uma experiência frustrante. As instruções parecem arbitrárias. Os registos parecem variáveis estranhas. Os endereços de memória parecem números mágicos. Nada faz sentido porque falta o contexto que justifica a existência de cada elemento.

O problema da ordem de aprendizagem

A forma como muitos estudantes abordam a informática está invertida. O percurso habitual tornou-se algo semelhante a: aprender Python, aprender JavaScript, aprender React, aprender Node.js, aprender Docker, aprender mais um framework e tentar aprender Assembly. Quando chegam ao último passo, sentem que entraram num mundo completamente diferente.

Mas talvez a sequência correta fosse mais próxima de: sistemas digitais, lógica booleana, arquitetura de computadores, organização de computadores, assembly, sistemas operativos, compiladores, linguagens de programação de alto nível e frameworks e ferramentas Nesta abordagem, cada novo conceito surge como consequência natural do anterior. Assembly deixa de parecer magia. Os sistemas operativos passam a resolver problemas concretos que o estudante já compreende. Os compiladores deixam de ser caixas negras. As linguagens de alto nível tornam-se abstrações compreensíveis em vez de ferramentas misteriosas.

A perda da cultura de engenharia

Existe também uma mudança cultural. Historicamente, a informática estava fortemente ligada à engenharia. Era esperado que um profissional compreendesse: como o hardware funciona, como a memória é organizada, como os programas são executados, como o sistema operativo gere recursos, como os protocolos de rede transportam informação. Hoje, uma parte significativa da indústria incentiva apenas o conhecimento necessário para produzir software rapidamente. Isto gera especialistas em ferramentas, mas não necessariamente especialistas em computação. Quando surge um problema fora da camada de abstração habitual, muitos profissionais ficam sem ferramentas conceptuais para o resolver.

O perigo das abstrações sem fundamentos

As abstrações existem para simplificar. No entanto, uma abstração só é verdadeiramente útil quando se compreende aquilo que está por baixo dela. Um programador pode utilizar memória dinâmica durante anos sem compreender conceitos como heap, stack, paginação, memória virtual,... Pode utilizar sockets sem compreender conceitos como TCP, IP, roteamento, controlo de congestionamento,... Pode utilizar threads sem compreender conceitos como escalonamento, sincronização, context switching,... O problema surge quando algo falha. Nesse momento, a abstração deixa de ser suficiente e os fundamentos tornam-se indispensáveis.

Aprender informática em vez de apenas programar

Programar é uma parte da informática. Importante, mas apenas uma parte. A informática inclui áreas fundamentais como sistemas digitais, arquitetura de computadores, sistemas operativos, redes, compiladores, bases de dados, teoria da computação, algoritmos, sistemas distribuídos,... Quem compreende estas áreas consegue adaptar-se facilmente a novas linguagens. O inverso nem sempre é verdade. Uma pessoa pode conhecer dez linguagens e ainda assim ter dificuldades em compreender como um computador realmente funciona.

Conclusão

O objetivo não é defender que todos os profissionais devam tornar-se especialistas em hardware ou em sistemas operativos. Mas é importante reconhecer que informática não é sinónimo de programação. As linguagens mudam. As frameworks desaparecem. As tecnologias entram e saem de moda. Os fundamentos permanecem.

Talvez o melhor conselho para quem começa hoje seja simples: Não estudes apenas linguagens. Estuda computadores. Quando compreendes o computador, as linguagens tornam-se apenas diferentes formas de comunicar com ele.
